## 변경 내용 (한 번에)

### 인프라
- `app/globals.css` — `.sr-only` 유틸리티 클래스
- `app/layout.tsx` — `<html lang="ko">`

### 모든 폼 (login, new post, comment) — 동일한 a11y 패턴 적용
- `useId()`로 input/error/root 각각 안정 ID 생성
- `<label htmlFor>` (보이는 라벨 또는 `.sr-only`)
- `aria-invalid`(이미 있던 것) + `aria-describedby`로 에러 메시지 연결
- 에러 `<p role="alert">` (필드별 + root 통합 에러)
- `<form aria-busy={isLoading}>`

### 동적 영역
- 로딩: `<p role="status" aria-live="polite">`
- 에러: `<p role="alert">`

### `app/posts/page.tsx`
- 행별 삭제 버튼: `aria-label='"<post.title>" 삭제'` — 같은 텍스트 버튼 여러 개를 스크린리더가 구분

## 평가

**성공.** 5개 폼 입력에 일괄 적용되었고, 06 실험에서 만든 패턴(`zodResolver` 폼 + `aria-invalid`)을 자연스럽게 확장. **새 폼을 만들 때 같은 패턴을 따르면 a11y가 자동 동반**되도록 코드 모양이 정착.

### 테스트 변경 영향

기존 58개 테스트는 모두 그대로 통과 (a11y 추가는 비파괴적). a11y 회귀 방지 테스트 1개 추가:

```ts
it('connects error message to textarea via aria-describedby (a11y)', ...)
```

→ aria-describedby가 가리키는 ID의 엘리먼트 텍스트가 실제 에러 메시지와 같은지 검증. 누군가 a11y 속성을 떼면 RED.

### 주목할 점

- **`useId()` 활용** — SSR/CSR 간 ID 충돌 없이 안정적인 폼 ↔ 에러 연결
- **React 19/Next 16** — `useId` 안정 동작 확인 (3개 폼 컴포넌트 모두)
- **i18n과 자연스러운 다음 단계** — 이제 모든 라벨/메시지가 명시 텍스트로 코드에 등장 → i18n 키 추출 대상이 깔끔하게 드러남

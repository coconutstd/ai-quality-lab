## 평가

**성공.** 42개 키, 43개 호출처를 한 번의 변경으로 propagate. UI에 한국어 하드코딩 0건.

### 함정 — Zod v4 message function (실시간으로 잡은 것)

처음엔 `z.email({ message: () => t(...) })` lazy 형태로 작성. 런타임은 통과했으나 **타입 정의가 좁아서** tsc strict가 7개 에러. 두 가지 트레이드오프:

| 옵션 | 장점 | 단점 |
|---|---|---|
| Lazy `() => t(...)` | locale 변경 즉시 반영 | 타입 cast 필요 |
| Eager `t(...)` (모듈 로드 시 평가) | 타입 깨끗 | locale 변경 후 schemas 모듈 reload 필요 |

→ 본 lab은 단일 locale 단일 세션 가정이므로 **Eager** 채택. 실제 다국어 SPA에서는 Lazy + cast 또는 [react-i18next의 `t()`를 컴포넌트 안에서 호출](https://react.i18next.com/) 패턴을 권장.

### 주목할 점

- **a11y와의 시너지** — 07에서 `<label>`을 도입한 직후 모든 라벨이 `t()` 추출 대상으로 깔끔하게 드러남. 추출 누락 가능성 ↓
- **컴파일 시점 안전성** — `t('typo.key')`는 즉시 TypeScript 에러. 카탈로그 동기화 책임이 사람에서 컴파일러로
- **production 마이그레이션 경로** — 본 lab의 in-house `t()`는 `next-intl` 또는 `react-i18next`의 `useTranslation()`으로 1:1 치환 가능. 키 카탈로그 자체는 그대로 사용
- **Server Component 호환** — `t()`는 순수 함수라 서버 컴포넌트에서도 동작. (locale을 cookie/header에서 읽도록 확장하면 SSR i18n도 동일 패턴)
